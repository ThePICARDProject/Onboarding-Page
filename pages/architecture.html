System Architecture

A bird's-eye view of how PICARD is put together. Read this first before diving into any individual layer.

PICARD is a three-layer system. Each layer owns a single concern, and the boundaries between them are deliberately narrow:

  • The presentation layer renders pages and collects user input.
  • The application layer persists state, queues work, and issues commands.
  • The execution layer spins up cluster resources and runs the actual algorithm.

The big architectural choice that shapes everything else is that the application layer doesn't talk to the cluster over an API — it shells out to bash scripts that wrap Docker and spark-submit. The cluster is treated as a fire-and-forget subprocess that writes results to shared disk. This is simpler than a real RPC layer but it's the source of most of PICARD's quirks (no callbacks, single-experiment-at-a-time, manual cleanup).

The three layers

Presentation

React 18 + Vite SPA on port 5173. React Router for navigation, TailwindCSS + MUI for UI, Axios for HTTP, and client-side parsing for JARs (JSZip) and CSVs (PapaParse). Auth state lives in localStorage and is gated by a ProtectedRoute wrapper.

Application

.NET 8 / C# API on port 5080. Controllers expose the REST surface, services hold the business logic, EF Core persists to MySQL/MariaDB. A System.Threading.Channels queue decouples HTTP submission from background execution; a hosted BackgroundService drains it one experiment at a time.

Execution

Docker Swarm running per-experiment Hadoop/Spark stacks. HDFS NameNode + YARN ResourceManager on the master node, DataNode + NodeManager on workers. Ephemeral containers inject the dataset and extract results. The whole stack is torn down after every run.

The full experiment lifecycle

Every experiment in PICARD takes the same path. Understanding this end-to-end flow is the single most useful thing you can learn on your first day.

  1. The user configures an experiment in the browser. They pick an algorithm (a .jar), a dataset (a .csv), set algorithm-specific parameters, and dial in cluster resources (node count, executor memory, cores, etc.). All of this happens in the React SPA on localhost:5173.
  2. The frontend POSTs the config to the backend. POST /api/experiment/submit goes through the Vite proxy to the .NET API on localhost:5080. The HTTP response returns immediately with an experiment ID — the actual work hasn't started yet.
  3. The backend persists the request and enqueues it. ExperimentService.SubmitExperimentAsync writes three rows in a single transaction (the experiment, its cluster params, and its algorithm param values), then drops the request into an unbounded Channel<ExperimentRequest>. Status: WaitingInQueue.
  4. A background worker dequeues the experiment. ExperimentWorkerService is a hosted BackgroundService running an infinite loop. It reads from the channel, opens a fresh DI scope (so EF Core gets a clean DbContext), and calls ExperimentService.RunExperimentAsync. Status flips to BeingExecuted.
  5. The DockerSwarm service launches a shell script. DockerSwarm rewrites docker-images/spark-hadoop/Dockerfile to bake in the user's algorithm JARs, then spawns submit-experiment.sh as a subprocess via System.Diagnostics.Process, passing ~20 positional arguments. The PID is persisted so the experiment can be killed.
  6. The script builds and deploys a Spark/Hadoop stack on Docker Swarm. It builds the image, deploys a master container and N worker containers wired together by a per-experiment overlay network, then runs an ephemeral dataset injector container that does hdfs dfs -put to load the CSV into HDFS.
  7. Spark runs the algorithm. The script invokes spark-submit --master yarn on the master container. Spark distributes the work to YARN-allocated executors on the worker nodes. Output is written back into HDFS at /data/{userId}/{experimentId}/.
  8. An ephemeral results extractor pulls the output back to disk. Another one-shot container runs hdfs dfs -getmerge to merge the distributed output files into a single result file at ./results/{userId}/{experimentId}/ on the host.
  9. The script tears down the stack. cleanup.sh runs unconditionally (success or failure) with a 30-second timeout, removing the Docker stack and cleaning up HDFS artifacts.
  10. The backend records the result and updates status. ExperimentService writes an ExperimentResult row pointing at the output file and transitions the experiment to Finished (or Failed with an error message truncated to 255 chars).
  11. The frontend has been polling and streaming the whole time. ResultsPage hits GET /api/experiment/status/{id} once a second and holds open a WebSocket to /experiment/watch/{id} for live log streaming (rendered as green-on-black terminal output). When the status flips to Finished, a download button appears.

Layer boundaries at a glance

These are the contracts that connect the layers. If you need to change the contract between two layers, both sides have to move together — otherwise stick to one layer at a time.

Boundary Mechanism Notes
Browser ↔ Presentation HTTP / WebSocket on localhost:5173 Vite dev server. /api/* is proxied to the backend.
Presentation ↔ Application REST + WebSocket on localhost:5080 Bearer token in Authorization header (set by AuthProvider). Logs streamed via WS.
Application ↔ Persistence EF Core → MySQL/MariaDB 13 DbSet<> entities. Code-first migrations. Serilog also writes structured logs to a Logs table.
Application → Execution System.Diagnostics.Process → bash One-way, fire-and-forget. No callbacks. Backend reads stderr and waits for exit code.
Execution → Application Shared local disk Results file at ./results/{userId}/{experimentId}/. Backend reads from disk after the script exits.
Execution internal Docker overlay network + HDFS + YARN Per-experiment overlay {stack}_cluster-network. Master container hostname is literally master.

A mental model that holds up

If you remember nothing else from this page, remember this:

The model

PICARD is a web app that runs bash scripts. The frontend is a thin shell over the backend. The backend is a REST API plus a single-threaded queue worker. The queue worker's entire job is to call a bash script with the right arguments. That bash script is what actually orchestrates the cluster — everything Docker- and Spark-related lives there, not in C#.

Once you internalize that, most of the system's behavior makes sense:

  • Why is execution sequential? Because the worker count is 1, and the DockerSwarm service rewrites a shared Dockerfile per experiment — running two at once would race on that file.
  • Why does cleanup fail open and need docker system prune? Because the cleanup path is a bash script with a 30-second timeout, not a transactional teardown.
  • Why does the backend not get notified when a job finishes? Because there is no callback — the backend just waits on the subprocess and reads the result file from disk.
  • Why is the parameter format for algorithms positional and sorted by DriverIndex? Because the bash script passes them as $1 $2 $3 ... to spark-submit.

Each layer has its own deep-dive page. Start with whichever one is closest to the work you've been asked to do.

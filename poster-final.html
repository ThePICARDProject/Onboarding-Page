Background

PICARD lets WVU researchers run machine-learning experiments on a dedicated Hadoop/Spark cluster without wiring up distributed infrastructure themselves. Upload an algorithm and dataset, pick parameters, hit submit: the platform handles auth, queueing, distributed execution, and result delivery.

Mission
Make distributed Big-Data experimentation accessible to any WVU researcher, regardless of systems background.

Origin: CoDRIFt

PICARD grew out of Dr. Devine's thesis on CoDRIFt, a Co-Training, Distributed, Random Incremental Forest tuned for rare-event detection (e.g. pulsars in radio-telescope records).

Running CoDRIFt once required stringing together 20+ desktop PCs by hand. PICARD productises that experience, and is algorithm-agnostic today. Any Scala JAR or Python script can run.

Current Capabilities

  • End-to-end web workflow: upload, configure, and run distributed experiments from the browser.
  • WVU-gated Google SSO restricts access to @mix.wvu.edu.
  • Spark-on-Hadoop execution across a dedicated Docker Swarm cluster.
  • Batch parameters: one algorithm, many parameter values, run sequentially.
  • Scala + Python 2.7 execution paths; live logs stream to the browser.

Known Limitations

Documentation gaps
Operational knowledge lives in graduates' heads: double-hop SSH, hand-crafted dotnet starts.
Single-experiment execution
Concurrent runs can delete another user's uploaded files; artefacts vanish.
Ungraceful failure
Crashed experiments orphan namenodes. Only known fix: docker system prune.
Long feedback loops
Misconfigured algorithms take ~5 min to fail: the Swarm must stand up a full cluster first.
Python version ceiling
The container's glibc pins Python at 2.7; modern ML libraries won't install.
Locked container state
A graduated student password-locked the core image. exec access requires a full rebuild.

NEW Parameter Sweep Engine
  • Matrix submissions: machine-learning research means comparing dozens of parameter variants. Describe a range once ("learning rate 0.001 to 0.1, step 0.01") and the backend expands it into every combination, queued as a single atomic batch. No more copy-paste resubmission for each variant.
  • Server-side orchestration: the whole batch is committed to the database before the first container starts, so a closed browser tab can't drop experiments mid-run.
  • Live batch monitor: a dedicated dashboard tracks per-variant progress in real time and supports cancellation when early results already tell the story.
  • Drops into the existing pipeline: each variant runs through the same queue and Spark cluster as a single experiment, preserving today's execution guarantees.

Parameter Sweep: How It Works

A researcher describes a search space once; the platform fans it out into every combination and runs them all.

01
Configure Ranges
Toggle any parameter into sweep mode and set start, step, and end values. A live preview shows exactly how many experiments will be queued before you commit.
02
Single Submission
One click sends the full range definition to the server. The backend computes every combination and writes the whole batch atomically, so a partial submission is impossible.
03
Sequential Execution
Each variant runs through the existing Spark cluster as a normal experiment. No new infrastructure required; no risk of variants interfering with each other.
04
Live Monitoring
A dedicated dashboard tracks every variant in real time. Cancel the batch at any point; results already completed are preserved.
Configure → Preview → Submit → Expand → Execute → Monitor

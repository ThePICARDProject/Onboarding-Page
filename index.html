PICARD · West Virginia University

Welcome to the PICARD project.

PICARD is a research platform that lets students and faculty run computationally heavy algorithms on a dedicated Hadoop/Spark cluster without having to wire up the infrastructure themselves. This portal is the fastest path from "I just joined" to "I'm shipping code."

Start the setup → Learn the architecture

What is PICARD?

PICARD provides a web-based interface to a Spark/Hadoop cluster running on dedicated hardware in a WVU building. Researchers upload their algorithms (as Scala JARs) and datasets (as CSVs), configure cluster resources, and submit experiments through a browser. The platform handles authentication, queueing, distributed execution, and result delivery.

The project was originally inspired by Dr. Devine's master's thesis on CoDRIFt, a custom RandomForest algorithm for astronomically large datasets. The original CoDRIFt work required wiring together 20+ PCs by hand to get acceptable performance — PICARD productizes that experience for anyone in the lab.

The three layers

PICARD is built as three loosely coupled layers. Each layer has a single, clear responsibility, and they communicate over well-defined boundaries. Click any card to dive in.

Presentation Layer

React + Vite single-page app on port 5173. Handles login, file uploads, experiment configuration, and real-time result streaming. This is what users see and click.

Application Layer

.NET 8 REST API on port 5080. Owns authentication, persistence (MySQL/MariaDB), the experiment queue, and orchestration of the cluster via shell scripts. The brain of the system.

Execution Layer

Docker Swarm running per-experiment Hadoop/Spark stacks. HDFS for storage, YARN for scheduling, Spark for the actual computation. Where algorithms actually run.

For a side-by-side view of all three layers and how they connect, see the architecture overview.

How a request flows through the system

At a high level, every experiment follows the same path. The frontend collects configuration and hands it to the backend, the backend persists it and queues it, a background worker dequeues it and launches a fresh Spark cluster on the swarm, results are written back to disk, and the frontend streams the logs and final output to the user's browser.

Browser → React (Vite, 5173) → .NET API (5080) → MySQL + Channel queue → DockerSwarm service → submit-experiment.sh → Spark on Hadoop/HDFS → Result file

Background

PICARD lets WVU researchers run machine-learning experiments on a dedicated Hadoop/Spark cluster without wiring up distributed infrastructure themselves. Upload an algorithm and dataset, pick parameters, hit submit — the platform handles auth, queueing, distributed execution, and result delivery.

Mission
Make distributed Big-Data experimentation accessible to any WVU researcher — regardless of systems background.

System Architecture

Presentation · React + Vite
SPA on port 5173. Google SSO gated to @mix.wvu.edu. Uploads, config, live logs.
↓
Application · .NET 8 API
Port 5080. Auth, MySQL persistence, experiment queue, and cluster orchestration.
↓
Execution · Docker Swarm
HDFS + YARN + Spark. One fresh cluster per submission — no cross-experiment state.
Browser → React → .NET API → Queue → Swarm → Spark → Result

Current Capabilities

  • End-to-end web workflow — upload, configure, and run distributed experiments from the browser.
  • WVU-gated Google SSO restricts access to @mix.wvu.edu.
  • Spark-on-Hadoop execution across a dedicated Docker Swarm cluster.
  • Batch parameters — one algorithm, many parameter values, run sequentially.
  • Scala + Python 2.7 execution paths; live logs stream to the browser.

Fall 2025 Handoff

Delivered by the preceding 481S cohort — the foundation we build on:

  • Python 2.7 execution path added to the Spark / Docker layers alongside the original Scala pipeline.

  • Hardened backend onboarding — credentials and cluster-access cleanup.

Known Limitations

Documentation gaps
Operational knowledge lives in graduates' heads — double-hop SSH, hand-crafted dotnet starts.
Single-experiment execution
Concurrent runs can delete another user's uploaded files — artefacts vanish.
Ungraceful failure
Crashed experiments orphan namenodes. Only known fix: docker system prune.
Long feedback loops
Misconfigured algorithms take ~5 min to fail — after the Swarm stands up a full cluster.
Python version ceiling
The container's glibc pins Python at 2.7; modern ML libraries won't install.
Locked container state
A graduated student password-locked the core image. exec access requires a full rebuild.

Going Forward

Fall 2026 attacks onboarding on two fronts: a first-class documentation portal for newcomers, and backend hardening so the system behaves predictably while they learn it.

DOC Onboarding Portal
  • Three-layer architecture walkthrough with per-layer deep dives.
  • Cold-start setup guide: SSH approval, ssh.wvu.edu tunnelling, starting the .NET backend.
  • Canonical troubleshooting reference — retiring folklore.
  • Published under ThePICARDProject to outlive any single cohort.
BE Backend Hardening
  • Graceful teardown of Spark / Hadoop services on crash.
  • Upgrade execution container to Python 3 with modern ML libraries.
  • Isolate per-user file storage so concurrent runs don't evict each other.
  • Submission-time parameter validation — fail fast, not after a 5-minute Swarm boot.
FE Frontend Polish
  • Structured error indicators on experiment failure; harden session handling, CSRF, and SSO token lifetime on the public deployment.

Origin — CoDRIFt

PICARD grew out of Dr. Devine's thesis on CoDRIFt — a Co-Training, Distributed, Random Incremental Forest tuned for rare-event detection (e.g. pulsars in radio-telescope records).

Running CoDRIFt once required stringing together 20+ desktop PCs by hand. PICARD productises that experience — and is algorithm-agnostic today. Any Scala JAR or Python script can run.
